Lisa — Developer Architecture Overview

Lisa is a chat-native autonomous work engine governed by constitutions, approvals, and sandbox isolation.

Lisa is designed to receive commands through Telegram, Slack, and WhatsApp only. There is no traditional dashboard. A user request enters chat, Lisa normalizes it, plans the work, reviews risk, routes models when needed, uses an isolated sandbox for risky execution, asks for approval at hard gates, audits every meaningful event, and reports back in chat.

InterfacesTelegram · Slack · WhatsApp
ControlJudicial Police + active Constitution
ExecutionSandboxRuntime only
Default sandbox512 MB · 1 CPU · no network

01 / End-to-end workflow

From chat command to governed result

Telegram / Slack / WhatsApp
→ MessageNormalizer
→ CommandParser
→ LisaCore
→ Judicial Police selects Constitution
→ Brain swarm plans and reviews
→ TokenBankBroker / ModelRouter if needed
→ CapabilityRegistry + Approval Gates
→ SandboxRuntime
→ AuditLog
→ Chat response
1

Chat webhook

Receives a platform payload from Telegram, Slack, or WhatsApp. Routes stay thin and should not contain business logic.

2

Normalize and parse

MessageNormalizer produces a common message shape. CommandParser detects request type, trigger, intent, and command arguments.

3

Mode selection

Judicial Police selects Normal, Autopilot, or Spike Constitution. Casual mentions of Autopilot must not activate Autopilot.

4

Plan and review

Planner creates steps. Feasibility finds blockers. Ranker scores readiness. Red Team Mirror checks adversarial risk.

5

Route tools and models

Teacher discovers tools when needed. TokenBankBroker and ModelRouter support complex reasoning and fallback routing.

6

Sandbox execution

WorkspaceRuntime must call SandboxRuntime. Risky commands must never run in the trusted backend process.

7

Approval gates

Principal approval is required for tool promotion, package install, secrets, env changes, merge, deploy, and database writes.

8

Audit and response

AuditLog records mode selection, verdicts, tool decisions, model usage, sandbox results, approvals, and final chat output.

02 / Brain role mapping

Every brain has a bounded responsibility

BrainResponsibilityBoundary
Planner BrainTurns the user goal into ordered steps, assumptions, risks, expected outputs, and approval checkpoints.Cannot execute tools directly.
Feasibility BrainChecks missing files, permissions, modules, infrastructure limits, sandbox constraints, and implementation blockers.Cannot silently approve risky work.
Ranker BrainScores clarity, safety, feasibility, rollback safety, token efficiency, and implementation readiness.Cannot bypass policy thresholds.
Teacher / Tool Discovery BrainResearches open-source tools, GitHub repos, packages, MCPs, SDKs, CLIs, and implementation options.Cannot install or trust tools directly.
Cyber-Immune BrainScans candidates for supply-chain risk, install scripts, permissions, prompt injection, and unsafe behavior.Cannot activate tools in trusted runtime.
Red Team MirrorAdversarially reviews plans, actions, tool choices, model routing, sandbox commands, and escalation paths.Cannot execute changes.
Principal / Auditor / 7th BrainFinal approval authority for risky evolution, tool promotion, policy changes, memory promotion, merge, and deploy.Cannot bypass Judicial Police.
Judicial PoliceTop-most enforcement authority. Applies the active Constitution and can allow, block, pause, stop, quarantine, or require approval.Cannot be disabled or bypassed.

03 / Operating modes

Different constitutions for different autonomy levels

Normal Constitution

Used for regular chat and /lisa plan. Plan-first, low-token, strict constraints, and no autonomous workspace execution by default.

Autopilot Constitution

Explicit trigger only. Allows brain swarm, larger token budgets, model switching, sandboxed patch/test workflows, and tool research.

Spike Constitution

Exploration only. Allows research, multi-model comparison, and sandbox prototypes. Blocks merge, deploy, secrets, and irreversible changes.

04 / General job router

Lisa classifies user work before acting

Job types

  • planning_job
  • research_job
  • implementation_job
  • tool_adoption_job
  • maintenance_job
  • security_job
  • autopilot_job
  • spike_job
  • unknown_job

Default behavior

Lisa should not ask unnecessary questions. It should make the safest plan, identify blockers, proceed if safe, and ask approval only when the next action is risky or impossible without user input.

05 / Tool adoption lifecycle

External capability is never trusted immediately

Teacher discovers candidate
→ ToolCandidate created
→ Cyber-Immune scans
→ Quarantine stores
→ SandboxRuntime tests
→ Red Team reviews
→ Principal approves or rejects
→ Judicial Police final verdict
→ Tool becomes trusted only if approved

Approval required for

  • Package install
  • MCP activation
  • CLI activation
  • Docker usage
  • External network access
  • Persistent storage
  • GitHub token usage
  • Any code-executing tool

ToolCandidate should record

  • Source URL, package name, license, maintainer signal
  • Install commands and required permissions
  • Network and filesystem behavior
  • Risk flags, trust score, reviews, and approval status

07 / Token and model routing

Autopilot uses capacity intentionally

TokenBankBroker

Checks FreeLLMAPI capacity, reserves tokens per task, records usage, and releases unused reservations.

ModelRouter

Selects model by task type, health, context requirement, JSON/tool support, latency target, and prior failures.

FallbackEngine

Switches model on timeout, rate limit, provider failure, empty output, JSON parse failure, or low confidence.

UsageLedger

Records model and token usage per task for auditing and future routing decisions.

08 / Approval and hard blocks

Powerful does not mean lawless

Approval required

  • Tool promotion, package install, MCP/CLI/Docker activation
  • Merge to main, deployment, env changes, secret access
  • Database writes or migrations
  • Large token-bank reservations
  • Sandbox network enablement or resource increase
  • Policy or constitution changes

Always blocked

  • Disabling Judicial Police, AuditLog, or SandboxRuntime
  • Bypassing approval gates or sandbox boundaries
  • Credential exfiltration or host secret access
  • Risky host command execution
  • Destructive command patterns: rm -rf, curl | sh, wget | sh, chmod 777, drop table, delete database
