01 / System overview

Developer mental model

Lisa should be read as an orchestration layer, not a static website and not a simple chatbot. Chat adapters receive messages. Core orchestration interprets the intent. Judicial Police applies the active Constitution. Brain modules plan and critique. Runtime modules handle tools, model routing, sandbox execution, persistence, notifications, and audit logging.

Core idea

Chat is the UI. Governance is the product.

Lisa must not depend on a traditional dashboard for user-facing control. Every meaningful workspace update should be explainable through chat: active brain, current decision, changed files/tools/workspaces, and what happens next.

Input boundary

Only Telegram, Slack, and WhatsApp should be treated as user-facing command interfaces.

Execution boundary

Risky commands, package installs, repo modifications, and tool trials belong in SandboxRuntime, not the trusted backend process.

Approval boundary

Secrets, env changes, deployment, database writes, tool promotion, and merge require explicit Principal approval.

Documentation boundary

Architecture and implementation notes belong in docs/. The site/ folder must remain the polished public developer overview.

03 / Seven-brain architecture

Each brain has one job and one boundary

The brains improve autonomy, but none of them bypass Judicial Police, AuditLog, SandboxRuntime, or Principal approval gates.

Planner Brain

Turns the user goal into ordered steps, assumptions, expected outputs, risk notes, rollback points, and approval checkpoints.

Boundary: cannot execute tools directly.
Feasibility Brain

Checks missing files, dependencies, permissions, infra limits, sandbox constraints, test availability, and implementation blockers.

Boundary: cannot silently approve risky work.
Ranker Brain

Scores clarity, safety, feasibility, rollback safety, token efficiency, confidence, and implementation readiness.

Boundary: cannot bypass thresholds.
Teacher Brain

Discovers packages, CLIs, SDKs, MCPs, examples, docs, GitHub repos, and integration options when the current toolset is insufficient.

Boundary: cannot install or trust tools.
Cyber-Immune Brain

Reviews supply-chain risk, install scripts, permissions, network behavior, suspicious prompts, dependency behavior, and unsafe execution paths.

Boundary: cannot promote tools alone.
Red Team Mirror

Adversarially reviews plans, commands, tool choices, model routing, escalation paths, and assumptions before sensitive actions.

Boundary: cannot execute changes.
Principal / Auditor

Final approval authority for risky evolution, tool promotion, memory promotion, env changes, deployment, merge, and policy changes.

Boundary: cannot disable Judicial Police.

04 / Constitutional operating modes

Autonomy changes by mode; enforcement does not

Normal

Plan-first mode

Used for regular chat and /lisa plan. Low token budget, strict constraints, no autonomous workspace execution by default.

  • Best for explanation and planning
  • Can inspect safe context
  • Requires escalation for risky actions
Autopilot

Governed autonomy

Explicit trigger only. Allows brain swarm, larger reasoning budget, sandboxed patch/test workflows, tool research, and approval-gated execution.

  • Must report meaningful updates
  • Must stop at hard gates
  • Must preserve audit trail
Spike

Safe exploration

Research and prototype mode. Allows comparison, sandbox trials, and recommendations while blocking irreversible production actions.

  • No merge or deploy
  • No secret access
  • No permanent trust promotion

SandboxRuntime

Owns risky execution: commands, package trials, repo modifications, and tool validation. Local simulators must be clearly labeled as simulated isolation.

ModelRouter

Selects models by task type, context need, health, latency, fallback policy, JSON/tool support, and budget constraints.

TokenBankBroker

Reserves, consumes, releases, and audits model capacity so Autopilot does not burn tokens blindly.

AuditLog

Records mode selection, brain verdicts, approval decisions, file changes, tool choices, sandbox results, and final chat output.

06 / Developer contracts

Rules future agents must preserve

Notification contract

Every meaningful environment/workspace update should explain the active brain, decision being made, changed file/tool/workspace, and next action.

Approval contract

Crucial updates require user validation before execution: merge, deploy, env, secret, DB write, policy change, tool trust, and sandbox escape risk.

No hardcoding contract

Models, provider fallback chains, severity thresholds, report limits, scanner behavior, and tunables belong in config/env/declarative settings.

Site protection contract

The site/ folder is a public-facing developer overview. Do not turn it into a status page again.

07 / Guardrails

Professional autonomy without chaos

Lisa should be aggressive about fixing broken layers, but conservative around irreversible actions. The correct behavior is inspect → plan → sandbox → test → report → ask approval when needed.

Always blocked

  • Disabling Judicial Police, AuditLog, or SandboxRuntime
  • Bypassing approval gates or sandbox boundaries
  • Credential exfiltration or trusted-host secret access
  • Replacing site/ with generated status output
  • Direct destructive commands without sandbox and approval

Approval required

  • Tool promotion from quarantine to trusted use
  • Package, MCP, CLI, or Docker activation
  • Merge to main, deployment, env changes, or external messages
  • Database writes, migrations, persistence changes, or secret usage
  • Any future edit to the site/ public landing page

Developer promise

Clear architecture. Safe autonomy. Mobile-readable docs. Desktop-grade detail.
