Chat-native autonomous agent system

Lisa turns chat commands into governed, restart-safe engineering work.

Lisa is controlled through Telegram, Slack, and WhatsApp. It receives a user command, creates a durable task, selects a constitutional mode, coordinates specialized brains, gates risky actions through approvals, executes only inside controlled runtimes, checkpoints progress, and reports every meaningful update back through chat.

→ IdempotencyKey + AuditEvent + Checkpoint
→ NotificationOutbox
→ Telegram / Slack / WhatsApp update

Webhook layer

Receives platform payloads, verifies configured secrets, normalizes input, creates durable tasks, and returns quickly.

Queue layer

Stores runnable work through DatabaseQueue by default or RedisQueue when configured. MemoryQueue is local/test only.

Worker layer

Claims tasks with leases, runs one bounded step, writes checkpoints, and recovers stale tasks after restarts.

Notification layer

Writes outgoing messages to NotificationOutbox first, then sends or simulated_sends with idempotency protection.

03 / Runtime and persistence

Restart-safe by design

Durable database

Tasks, long-running sessions, brain messages, working memory snapshots, checkpoints, approvals, audit events, token usage, tool quarantine, leases, idempotency keys, and notifications are database-backed.

Recovery service

If a worker dies, stale leases expire. Lisa reloads the latest checkpoint, restores working memory, queues continuation, and reports recovery through chat.

Replay guard

External effects such as notifications, approvals, model calls, sandbox runs, token reservations, and final responses use idempotency keys to prevent duplicates.

Normal Mode

Plan-first, low autonomy, no risky execution unless approved.

Autopilot Mode

Explicit trigger only, bounded iterations, checkpoints, budget limits, approval pauses.

Spike Mode

Exploration and prototypes only; no irreversible production changes.
